# anight Core Philosophy

## Core Is Specification, Not Code
The `core/` directory is the immutable specification of anight.
It defines what the system is, not how a specific runtime executes it.

## Three-Layer Boundary
- `core/`: immutable, declarative contracts and role definitions.
- `adapters/`: replaceable implementations that consume `core/`.
- `workspace/`: runtime data generated by executions.

## Immutability Rules
- Everyone may read `core/`.
- No one should modify `core/` without explicit architectural review.
- Any change in `core/` is a philosophy-level change, not a simple implementation change.

## Core Eligibility Criteria
A file belongs in `core/` only if all conditions hold:
1. It is declarative text (Markdown, YAML, JSON, or plain text).
2. It does not depend on any specific programming language, runtime, framework, or LLM provider.
3. A reader can understand and re-implement the system from this file alone.

## Governance
Treat updates under `core/` as architecture proposals.
Apply stricter review than normal implementation changes.
See `core/READONLY` for the read-only marker.
